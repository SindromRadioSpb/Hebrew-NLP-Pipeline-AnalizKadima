 Блок 1: NLP Stack — Transformer Backbone

 ────────────────────────────────────────────────────────────────────────────────

 ### R-1.1 spacy-transformers в core dependencies

 Сейчас: spacy-transformers отсутствует в pyproject.toml. Установка pip install kadima не ставит его.

 Что даёт: Фундамент. spacy-transformers — это мост между spaCy и любыми HuggingFace-моделями (включая NeoDictaBERT).
 Без этого пакета spaCy физически не может загружать transformer models и прокидывать embeddings в pipeline.

 Расширенность: Нулевая сама по себе — это просто зависимость. Но без неё R-1.2–R-1.4 невозможны. Это как установить
 драйвер видеокарты — сам по себе драйвер ничего не делает, но без него GPU не работает.

 Качество: Не влияет напрямую. Влияет через последующие пункты.

 ────────────────────────────────────────────────────────────────────────────────

 ### R-1.2 NeoDictaBERT transformer component

 Сейчас: nlp/pipeline.py делает spacy.blank("xx") — пустой pipeline без модели. Токены не имеют векторов. M5 (NP
 chunker) и M9 (NER) работают на голых правилах.

 ⁧Что делает NeoDictaBERT: Берёт каждый токен ивритского текста, прогоняет через 12-слойный transformer и выдаёт вектор 
 768 чисел — контекстное представление слова. «קל» в предложении «קל ללימוד» (лёгкий для изучения) получает другой
 вектор, чем «קל» в «שמעתי קל» (я слышал голос). Одинаковое написание — разный смысл — разные векторы.⁩                  ───────────────────────────────────────────────────────────────────────────────────────────────────────────────────────

 Что конкретно даёт:

 ┌──────────────┬─────────────────────────────────────────────────────────┐
 ┌─────────────────────┬──────────────────────────────────────────────────────────────────────────────────────────────┐
 │ До (правила)        │ После (NeoDictaBERT)                                                                         │
 ├─────────────────────┼──────────────────────────────────────────────────────────────────────────────────────────────┤
 │ ⁧                    │ NP chunking по шаблонам: «если NOUN + NOUN — это chunk»                                      │ └──────────────┴─────────────────────────────────────────────────────────┘
 ├─────────────────────┼──────────────────────────────────────────────────────────────────────────────────────────────┤
 │ NER по словарю имён │ NER по контексту: «Бен-Гурион» в предложении про историю → PERSON, в «аэропорт Бен-Гурион» → │
 │                     │ LOCATION                                                                                     │
 ├─────────────────────┼──────────────────────────────────────────────────────────────────────────────────────────────┤
 │ ⁧                    │ KB search по точному совпадению строк                                                        │ ───────────────────────────────────────────────────────────────────────────────────────────────────────────────────────
 ├─────────────────────┼──────────────────────────────────────────────────────────────────────────────────────────────┤
 │ ⁧                    │ Term disambiguation невозможно                                                               │ ⠇ streaming • 0s | connected
 └─────────────────────┴──────────────────────────────────────────────────────────────────────────────────────────────┘

 Расширенность: Максимальная. Это единственное изменение, которое даёт системе семантическое понимание вместо
 поверхностного pattern matching. Все модули, которые используют embeddings (M5, M9, KB, clustering), начинают работать
 качественно.

 Качество: Трансформационный скачок. NP chunking и NER переходят из категории «работает, но грубо» в
 «конкурентоспособно с коммерческими решениями».

 ────────────────────────────────────────────────────────────────────────────────

 ### R-1.3 Обновление pipeline builder

 Сейчас: build_pipeline() всегда создаёт пустой spacy.blank("xx") и добавляет два HebPipe-компонента. Нет выбора —
 использовать transformer или нет.

 Что делает: Добавляет параметр use_transformer=True/False.

 ```
   use_transformer=True  →  [NeoDictaBERT] → [HebPipe SentSplit] → [HebPipe Morph] → [Ngram] → [NPChunk] → [Noise]
   use_transformer=False →  [HebPipe SentSplit] → [HebPipe Morph] → [Ngram] → [NPChunk] → [Noise]
 ```

 Что конкретно даёт:

 ┌───────────────────┬────────────────────────────────────────────────────────────────────────────────────────────────┐
 │ Аспект            │ Эффект                                                                                         │
 ├───────────────────┼────────────────────────────────────────────────────────────────────────────────────────────────┤
 │ Гибкость          │ На машине с 8 GB VRAM — True, на ноутбуке без GPU — False. Один код, два режима                │
 ├───────────────────┼────────────────────────────────────────────────────────────────────────────────────────────────┤
 │ Модульность       │ Компоненты M5 и M9 сами проверяют: «Есть ли в doc векторы? Если да — использую embeddings,     │
 │                   │ если нет — fallback на правила»                                                                │
 ├───────────────────┼────────────────────────────────────────────────────────────────────────────────────────────────┤
 │ Тестируемость     │ Тесты pipeline можно прогонять в режиме False (быстро, без GPU), а полный прогон с True —      │
 │                   │ отдельно                                                                                       │
 ├───────────────────┼────────────────────────────────────────────────────────────────────────────────────────────────┤
 │ Конфигурируемость │ config.yaml → use_transformer: true/false — пользователь решает без правки кода                │
 └───────────────────┴────────────────────────────────────────────────────────────────────────────────────────────────┘

 Расширенность: Средняя. Не даёт нового функционала, но делает архитектуру гибкой. Без этого пункта transformer либо
 всегда включён (ломает ноутбуки без GPU), либо всегда выключен (NeoDictaBERT бесполезен).

 Качество: Улучшает developer experience и deployment flexibility.

 ────────────────────────────────────────────────────────────────────────────────

 ### R-1.4 Graceful degradation

 Сейчас: Если NeoDictaBERT не загрузился (нет VRAM, нет интернета для скачивания, модель повреждена) — pipeline падает
 с ошибкой.

 Что делает: При неудачной загрузке transformer → warning в лог → автоматический fallback на spacy.blank → pipeline
 работает без embeddings.

 ```
   try:
       nlp = spacy.load("he_kadima_with_trans former")
   except Exception:
       logger.warning("NeoDictaBERT unavailable, falling back to rule-based pipeline")
       nlp = build_pipeline(use_transformer=F alse)
 ```

 Что конкретно даёт:

 ┌──────────────────────────────────────┬──────────────────────────┬──────────────────────────────────────────────────┐
 │ Ситуация                             │ Без graceful degradation │ С graceful degradation                           │
 ├──────────────────────────────────────┼──────────────────────────┼──────────────────────────────────────────────────┤
 │ GPU занят другой задачей             │ 💥 Pipeline crashed      │ ⚠️ Warning + работает без embeddings             │
 ├──────────────────────────────────────┼──────────────────────────┼──────────────────────────────────────────────────┤
 │ Модель не скачана                    │ 💥 Pipeline crashed      │ ⚠️ Warning + работает без embeddings             │
 ├──────────────────────────────────────┼──────────────────────────┼──────────────────────────────────────────────────┤
 │ VRAM исчерпан (8 GB, другая модель)  │ 💥 OOM error             │ ⚠️ Warning + работает без embeddings             │
 ├──────────────────────────────────────┼──────────────────────────┼──────────────────────────────────────────────────┤
 │ Первый запуск (модель ещё не         │ 💥 Pipeline crashed      │ ⚠️ Warning + работает, параллельно качается      │
 │ скачана)                             │                          │ модель                                           │
 └──────────────────────────────────────┴──────────────────────────┴──────────────────────────────────────────────────┘

 Расширенность: Не даёт нового функционала. Но делает систему production-ready. Без этого — один сбой железа убивает
 весь pipeline. С этим — система деградирует грациозно.

 Качество: Критически важна для reliability. Разница между «можно использовать в проде» и «лабораторный прототип».

 ────────────────────────────────────────────────────────────────────────────────

 Связка: как это работает вместе

 ```
   R-1.1 (зависимость)
       ↓
   R-1.2 (компонент)
       ↓
   R-1.3 (гибкий pipeline)  ←  R-1.4 (страховка)
       ↓
   M5 (NP chunker на embeddings)   — Tier 2
   M9 (NER на embeddings)          — Tier 2
   KB (embedding search)           — Tier 2
   Term clustering                 — Tier 2
 ```

 R-1.1 — фундамент (зависимость)
 R-1.2 — ядро (семантическое понимание текста)
 R-1.3 — гибкость (можно вкл/выкл)
 R-1.4 — надёжность (не падает при сбоях)

 Без R-1.2 — вся система работает на правилах (pattern matching), как сейчас. Это работает, но это surface-level
 анализ: «видит шаблоны, не понимает смысл».

 С R-1.2 — система получает контекстное понимание: одинаковые слова в разных контекстах = разные значения. Это то, что
 отличает «NLP pipeline» от «string matching с красивой обёрткой».

Блок 2: NP Chunker (M5) — Transformer-based

 ────────────────────────────────────────────────────────────────────────────────

 ### R-2.1 Переписать M5 на transformer embeddings

 Сейчас: engine/np_chunker.py (105 строк) работает по правилам: «если NOUN + NOUN рядом — это chunk», «если ADJ + NOUN
 — это chunk». Это работает для простых конструкций, но ломается на сложных.

 Что делает transformer embeddings: NeoDictaBERT даёт вектор 768 чисел для каждого токена. NP Chunker берёт векторы
 токенов в окне, прогоняет через классификатор (линейный слой или spaCy Tok2Vec → reduce_mean pooling → output), и
 классификатор решает: «граница NP здесь?»

 ```
   ⁧Правила (сейчас):    "פלדה אל-חלד" → ❌ два отдельных NP (разные существительные)⁩                                   
   ⁧Transformer:          "פלדה אל-חלд" → ✅ один NP (эмбеддинги показывают семантическую связь)⁩                        
 ```

 Что конкретно даёт:

 ┌──────────┬─────────────────────────────────────────┬────────────────────────────┐
 │ Ситуация │ Правила (сейчас)                        │ Transformer (после R-2.1)  │
 ├──────────┼─────────────────────────────────────────┼────────────────────────────┤
 │ ⁧         │ Простой ADJ+NOUN: "בית גדול"            │ ✅ Работает                │
 ├──────────┼─────────────────────────────────────────┼────────────────────────────┤
 │ ⁧         │ Смихута (construct state): "חוזק מתיחה" │ ✅ Работает (оба NOUN)     │
 ├──────────┼─────────────────────────────────────────┼────────────────────────────┤
 │ ⁧         │ Имена собственные: "אפרים קישון"        │ ❌ Два отдельных NP        │
 ├──────────┼─────────────────────────────────────────┼────────────────────────────┤
 │ ⁧         │ Термины с предлогами: "סגסוגת של פלדה"  │ ❌ "של" ломает шаблон      │
 ├──────────┼─────────────────────────────────────────┼────────────────────────────┤
 │ ⁧         │ Сложные NP: "תוצאות הבדיקה הסופית"      │ ⚠️ Частично                │                                    ───────────────────────────────────────────────────────────────────────────────────────────────────────────────────────
 ├──────────┼─────────────────────────────────────────┼────────────────────────────┤
 │ ⁧         │ Ложные срабатывания: "הבית והוא הלך"    │ ⚠️ "הבית" может быть chunk │                                   
 └──────────┴─────────────────────────────────────────┴────────────────────────────┘

 Расширенность: Высокая. NP chunking — это вход для term extraction (M8). Лучше чанки → лучше термины → лучше словарь.
 Один улучшенный модуль тянет за собой качество всего downstream pipeline.

 Качество: Переход из «работает на 70% случаев» в «работает на 90%+ случаев».

 ────────────────────────────────────────────────────────────────────────────────

 ### R-2.2 spaCy config.cfg с transformer model

 Сейчас: Pipeline собирается программно в nlp/pipeline.py через spacy.blank("xx") + add_pipe(). Нет формального
 описания конфигурации pipeline.

 Что делает config.cfg: Формальный файл spaCy, который описывает весь pipeline: какие компоненты, в каком порядке,
 какие модели, какие гиперпараметры. Это стандарт spaCy для reproducible pipelines.

 ```ini
   # Что будет в config.cfg:
   [nlp]
   pipeline = ["kadima_transformer", "kadima_sent_split", "kadima_morph",
               "ngram_extractor", "np_chunker", "noise_classifier"]

   [components.kadima_transformer]
   factory = "kadima_transformer"

   [components.kadima_transformer.m odel]
   name = "dicta-il/neodictabert"
 ```

 Что конкретно даёт:

 ┌──────────────────┬─────────────────────────────────────────────┬───────────────────────────────────────────────────┐
 │ Аспект           │ Без config.cfg                              │ С config.cfg                                      │
 ├──────────────────┼─────────────────────────────────────────────┼───────────────────────────────────────────────────┤
 │ Reproducibility  │ Настройки разбросаны по коду                │ Один файл, воспроизводимый pipeline               │
 ├──────────────────┼─────────────────────────────────────────────┼───────────────────────────────────────────────────┤
 │ spacy train      │ Невозможно (нет config)                     │ ✅ Можно дообучать NER через spacy train          │
 │                  │                                             │ config.cfg                                        │
 ├──────────────────┼─────────────────────────────────────────────┼───────────────────────────────────────────────────┤
 │ spacy debug data │ Невозможно                                  │ ✅ Проверка совместимости данных и pipeline       │
 ├──────────────────┼─────────────────────────────────────────────┼───────────────────────────────────────────────────┤
 │ Version control  │ Код меняется, неясно что поменялось в       │ Config.cfg в git — видно каждое изменение         │
 │                  │ pipeline                                    │ конфигурации                                      │
 ├──────────────────┼─────────────────────────────────────────────┼───────────────────────────────────────────────────┤
 │ Документация     │ «Посмотри код»                              │ «Посмотри config.cfg»                             │
 └──────────────────┴─────────────────────────────────────────────┴───────────────────────────────────────────────────┘

 Расширенность: Средняя. Не даёт нового функционала пользователю. Но открывает путь к spacy train (дообучение NER) и
 делает pipeline declarative вместо imperative.

 Качество: Улучшает developer experience и открывает ML-возможности (training).

 Блок 3: NER (M9/M17) — NeoDictaBERT backbone

 ────────────────────────────────────────────────────────────────────────────────

 ### R-3.1 Добавить NeoDictaBERT backend для NER

 Сейчас: engine/ner_extractor.py (314 строк) имеет два backend: heq_ner (внешняя модель) и rules (паттерны).
 NeoDictaBERT не используется.

 Что делает NeoDictaBERT для NER: Вместо внешней модели или словарей — transformer сам «смотрит» на контекст и решает:
 «Бен-Гурион» в «премьер-министр Бен-Гурион» → PERSON, в «аэропорт Бен-Гурион» → LOCATION. Одинаковое написание —
 разный контекст — разные метки.

 Что конкретно даёт:

 ┌───────────────────┬────────────────────────────┬───────────────────────────────┬───────────────────────────────────┐
 │ Аспект            │ HeQ-NER (сейчас)           │ Rules (сейчас)                │ NeoDictaBERT (после R-3.1)        │
 ├───────────────────┼────────────────────────────┼───────────────────────────────┼───────────────────────────────────┤
 │ Тип модели        │ Предобученная NER-модель   │ Словари + regex               │ Contextual transformer            │
 ├───────────────────┼────────────────────────────┼───────────────────────────────┼───────────────────────────────────┤
 │ Качество          │ Высокое (предобученная)    │ Низкое (только известные      │ Высокое + контекстное понимание   │
 │                   │                            │ сущности)                     │                                   │
 ├───────────────────┼────────────────────────────┼───────────────────────────────┼───────────────────────────────────┤
 │ Новые сущности    │ ❌ Только то, на чём       │ ❌ Только то, что в словаре   │ ✅ Обобщает на новые контексты    │
 │                   │ обучена                    │                               │                                   │
 ├───────────────────┼────────────────────────────┼───────────────────────────────┼───────────────────────────────────┤
 │ Доменная          │ Нет (общая модель)         │ Ручное добавление             │ ✅ Дообучение на своих данных     │
 │ адаптация         │                            │                               │                                   │
 ├───────────────────┼────────────────────────────┼───────────────────────────────┼───────────────────────────────────┤
 │ VRAM              │ ~500 MB                    │ 0                             │ ~500 MB (использует backbone из   │
 │                   │                            │                               │ R-1.2)                            │
 ├───────────────────┼────────────────────────────┼───────────────────────────────┼───────────────────────────────────┤
 │ Скорость          │ Средняя                    │ Быстрая                       │ Средняя                           │
 └───────────────────┴────────────────────────────┴───────────────────────────────┴───────────────────────────────────┘

 Расширенность: Высокая. NER → term candidates → term extraction. Лучше NER → лучше кандидаты в термины. Plus:
 именованные сущности — отдельный продукт (можно показывать пользователю «кто/что упоминается в корпусе»).

 Качество: Главное — дообучаемость. HeQ-NER — чёрный ящик. NeoDictaBERT — можно дообучить на своих данных
 (материаловедение, юриспруденция, медицина). Это превращает KADIMA из «общего инструмента» в «доменный инструмент».

 ────────────────────────────────────────────────────────────────────────────────

 ### R-3.2 NER training data pipeline (Label Studio → spaCy)

 Сейчас: annotation/ml_backend.py существует, но экспорт аннотаций из Label Studio в формат для обучения не реализован
 (или реализован минимально).

 Что делает closed loop:

 ```
   Шаг 1: KADIMA pipeline прогоняет текст → находит NER-сущности
            ↓
   Шаг 2: Pre-annotations отправляются в Label Studio (JSON import)
            ↓
   Шаг 3: Аннотатор проверяет/корректирует в LS UI
            ↓
   Шаг 4: Экспорт из LS → CoNLL-U формат (реализуется в R-3.2)
            ↓
   Шаг 5: CoNLL-U → spaCy training data → `spacy train` на NeoDictaBERT
            ↓
   Шаг 6: Улучшенная модель → лучше pre-annotations → шаг 1
 ```

 Что конкретно даёт:

 ┌────────────────────┬──────────────────────────────────────┬───────────────────────────────────────────────────┐
 │ Аспект             │ Без training pipeline                │ С training pipeline                               │
 ├────────────────────┼──────────────────────────────────────┼───────────────────────────────────────────────────┤
 │ Качество NER       │ Фиксированное (предобученная модель) │ Улучшается с каждой аннотацией                    │
 ├────────────────────┼──────────────────────────────────────┼───────────────────────────────────────────────────┤
 │ Доменная адаптация │ ❌                                   │ ✅ Аннотатор размечает свой домен → модель учится │
 ├────────────────────┼──────────────────────────────────────┼───────────────────────────────────────────────────┤
 │ Затраты аннотатора │ Высокие (размечает с нуля)           │ Низкие (исправляет pre-annotations)               │
 ├────────────────────┼──────────────────────────────────────┼───────────────────────────────────────────────────┤
 │ Обратная связь     │ Нет                                  │ Есть: аннотация → обучение → лучше предсказания   │
 └────────────────────┴──────────────────────────────────────┴───────────────────────────────────────────────────┘

 — NER адаптируется к любому домену через аннотации.

 Качество: Превращает NER из «one-shot модель» в «continuously learning система».

 Блок 4: KB Entity Linking — Embedding-based

 ────────────────────────────────────────────────────────────────────────────────

 ### R-4.1 KB entity linking через NeoDictaBERT embeddings

 ⁧Сейчас: kb/search.py ищет по точному совпадению строк: запрос «פלדה» → находит только записи с точным текстом «פלדה». 
 «סגסוגת פלדה» или «מתכת פלדה» — не найдёт.⁩                                                                            

 Что делает embedding search: Каждый термин в KB хранит NeoDictaBERT-вектор. При поиске система считает cosine
 similarity между вектором запроса и векторами всех терминов, возвращает top-K наиболее похожих.

 ```
   ⁧Запрос: "פלדה" (сталь)⁩                                                                                             

   Точное совпадение (сейчас):
   ⁧  → только "פלדה"⁩                                                                                                  

   Embedding search (после R-4.1):
   ⁧  → "פלדה"        (similarity: 1.00)⁩                                                                               
   ⁧  → "סגסוגת"      (similarity: 0.87)  — сплав⁩                                                                      
   ⁧  → "מתכת"        (similarity: 0.8⁩                                                                                    ⁧  → "מתכת"        (similarity: 0.82)  — металл⁩                                                                       ``
   ⁧  → "אלומיניום"    (similarity: 0.75)  — алюминий⁩                                                                     streaming • 2s | connected
   ⁧  → "נחושת"        (similarity: 0.71)  — медь⁩                                                                        ──────────────────────────────────────────────────────────────────────────────────────────────────────────────────────
 ```

 Что конкретно даёт:

 ┌──────────────────────────────┬────────────────────────────┬────────────────────────────────────────┐
 │ Аспект                       │ Точное совпадение (сейчас) │ Embedding search (после R-4.1)         │
 ├──────────────────────────────┼────────────────────────────┼────────────────────────────────────────┤
 │ Находит только точные строки │ ✅                         │ ✅                                     │
 ├──────────────────────────────┼────────────────────────────┼────────────────────────────────────────┤
 │ Находит синонимы             │ ❌                         │ ✅ (векторы близких слов близки)       │
 ├──────────────────────────────┼────────────────────────────┼────────────────────────────────────────┤
 │ Находит связанные термины    │ ❌                         │ ✅ («сталь» → «сплав», «металл»)       │
 ├──────────────────────────────┼────────────────────────────┼────────────────────────────────────────┤
 │ Обработка опечаток           │ ❌                         │ ⚠️ Частично (близкие векторы)          │
 ├──────────────────────────────┼────────────────────────────┼────────────────────────────────────────┤
 │ Масштаб                      │ 10K терминов — быстро      │ 10K терминов — быстро (cosine = numpy) │
 └──────────────────────────────┴────────────────────────────┴────────────────────────────────────────┘

 Расширенность: Высокая. Это превращает KB из «телефонной книги» в «семантическую сеть». Пользователь может исследовать
 связи между терминами, находить кластеры, обнаруживать пропущенные связи.

 Качество: Entity linking — основа терминологических баз. Без embeddings KB — просто SQL-таблица. С embeddings —
 интеллектуальный инструмент.

 ### R-5.1 Миграция с прямого sqlite3 на SQLAlchemy

 Сейчас: data/db.py использует sqlite3.connect(), прямые SQL-запросы через cursor.execute(). Нет type safety, нет ORM,
 нет connection pooling.

 ```python
   # Сейчас:
   conn = get_connection()
   cursor = conn.cursor()
   cursor.execute("SELECT * FROM corpora WHERE id = ?", (corpus_id,))
   row = cursor.fetchone()
   corpus = {"id": row[0], "name": row[1], ...}  # ручная распаковка
 ```

 ```python
   # После SQLAlchemy:
   session = get_session()
   corpus = session.get(Corpus, corpus_id)  # type-safe объект
   print(corpus.name)  # IDE autocomplete работает
 ```

 ├─────────────────────────┼────────────────────────────────────┼────────────────────────────────────────────┤

 ├─────────────────────────┼────────────────────────────────────┼────────────────────────────────────────────┤

 Сейчас: Нет модуля. YAKE уже в optional deps.

 Что делает: Извлечение ключевых фраз (не терминов). YAKE — unsupervised алгоритм: считает статистики слов (позиция в
 тексте, частота, контекст) и выдаёт ranked keyphrases.

 Отличие от M8 Term Extractor:

 ┌────────────────────┬──────────────────────────┬─────────────────────────────────────┐
 │ Аспект             │ M8 Term Extractor        │ M24 Keyphrase Extractor             │
 ├────────────────────┼──────────────────────────┼─────────────────────────────────────┤
 │ Подход             │ Linguistic (POS, NP, AM) │ Statistical (TF, position, context) │
 ├────────────────────┼──────────────────────────┼─────────────────────────────────────┤
 │ Домен              │ Доменные термины         │ Общие ключевые фразы                │
 ├────────────────────┼──────────────────────────┼─────────────────────────────────────┤
 │ Нужен gold corpus? │ Да (для валидации)       │ Нет (unsupervised)                  │
 └────────────────────┴──────────────────────────┴─────────────────────────────────────┘

 Что конкретно даёт: Быстрый обзор «о чём этот текст» без запуска полного pipeline.
